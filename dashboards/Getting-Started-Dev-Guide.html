1 The one big idea: GitHub is your cloud

The single mental shift that makes everything else click: your code lives on your Mac, and GitHub is its cloud backup — not iCloud, not Dropbox.

Git + GitHub already give you backup, a full history of every change, and sync across machines — the same things iCloud does for documents, but purpose-built for code and standard across the industry. So you don't need a sync service on top; it's redundant and it actively causes problems.

2 Your folder setup

One home for everything you build: ~/Developer. macOS even gives it a special icon, and it's never touched by iCloud.

The golden rule: 1 project = 1 folder = 1 git repo = 1 GitHub repo. Never nest one repo inside another.

~/Developer/
├── HTML-Dashboards/        # a repo → GitHub → Vercel
│   ├── .git/               # the history (git's, not yours to touch)
│   ├── index.html
│   └── dashboards/
└── Literature-Tracker/     # your second project, its own repo
    └── .git/

Move your projects out of iCloud (one time)

Do this after each project is committed and pushed, so GitHub holds a safe copy first:

mkdir -p ~/Developer
mv ~/Documents/Coding\ Projects/HTML-Dashboards   ~/Developer/
mv ~/Documents/Coding\ Projects/Literature-Tracker ~/Developer/
Naming

Name folders in lowercase-with-hyphens (e.g. literature-tracker) — the same tidy convention you already use for your dashboard files. Moving a folder won't affect Vercel: it deploys from GitHub, not from your Mac.

3 Starting a project with gh

You're signed in to the GitHub CLI (gh) as @pmjzzvs89z, so creating and connecting a repo is one command. Which one depends on whether code already exists.

A · Brand-new project — let the clone create the folder

cd ~/Developer
gh repo create my-project --private --clone --add-readme
cd my-project          # already wired to GitHub — just start coding

Best default: the folder is born connected, so you can never forget to link it.

B · Code already exists in a folder

cd ~/Developer/my-project
git init -b main                       # skip if it's already a repo
git add -A && git commit -m "Initial commit"
gh repo create my-project --private --source=. --remote=origin --push

That last line creates the GitHub repo from your folder, names the remote origin, and pushes — all at once. Swap --private for --public to make it world-readable (Vercel deploys either).

What "connected" actually means

git remote -v              # the link:     origin → github.com/you/repo
git status -sb | head -1   # the tracking: ## main...origin/main

remote = where to push. tracking = so a bare git push knows where to go without arguments.

4 The daily loop

Three moves, every working session. Think of push as "save to the cloud."

git pull                              # START — grab the latest (safe even solo)
# … make your changes …
git add -A
git commit -m "Add contact form"      # say WHAT changed, present tense
git push                              # END — back it up to GitHub
Your shortcut with Claude Code

For your dashboards, that whole end-of-session step is one word: drop the file in dashboards/ and say "publish." Claude rebuilds the index, commits, and pushes — and Vercel deploys.

Writing a good commit message

  • Small & often beats one giant commit — each is a save point you can return to.
  • Present tense, what changed: "Fix mobile layout", not "fixed stuff".
  • If why isn't obvious, add a second line explaining it.

5 Deploying with Vercel

Your Vercel project is connected to your GitHub repo, so there's nothing manual to "deploy."

  • Push to main = live. Every push auto-builds and updates html-dashboards.vercel.app in ~30–60s.
  • What goes up: everything committed to the repo — not just HTML. Static files are served as-is at their path.
  • Watch a build: Vercel dashboard → your project → Deployments.
Committed = public

On a public deploy, any file you commit is fetchable by URL — even ones nothing links to (e.g. a stray notes file becomes your-site.vercel.app/notes.txt). Commit deliberately. This leads straight to the next section.

6 Secrets & safety

The one rule that saves real money and headaches: never commit secrets. API keys, passwords, tokens, .env files — none of it belongs in git.

Keep them out with .gitignore

Put a .gitignore in every repo. Anything listed is invisible to git, so it never gets committed or deployed:

# .gitignore
.env
.env.local
*.key
.DS_Store
node_modules/
If you ever commit a secret

Removing it from git isn't enough — it stays in the history. Rotate the key (regenerate it at the provider) immediately, then stop tracking the file:

git rm --cached .env                 # stop tracking it (keeps your local copy)
echo ".env" >> .gitignore
git commit -m "Stop tracking .env"
# …then ROTATE the exposed key at the provider — assume it's compromised.

7 Good git habits & branching

  • main stays shippable. Since main auto-deploys, keep it in a working state.
  • Commit at every "save point" — a finished thought, a working feature, before a risky change.
  • Push when you step away so your work is backed up off your laptop.

Branch when you want to experiment safely

A branch is a private sandbox — play without touching the live site, then merge when happy. For solo static sites, committing straight to main is fine; reach for a branch when a change is big or risky.

git switch -c try-new-header     # branch off main
# … experiment, commit …
git push -u origin try-new-header
gh pr create --fill              # open a pull request to review & merge
git switch main && git pull       # back to main once it's merged

8 Backups & undo

Once it's pushed, it's safe — GitHub is your backup. If your laptop vanished, git clone brings every pushed project back. And almost anything can be undone:

git restore FILE                 # discard unsaved changes to one file
git restore .                    # discard ALL unsaved changes (careful)
git restore --staged FILE        # unstage a file (keeps your edits)
git commit --amend -m "Better message"   # fix the last message (if not pushed)
git revert HASH                  # undo a pushed commit — safely, as a new commit
git restore --source HASH FILE   # bring an old version of a file back
Find a HASH

git log --oneline lists recent commits with their short hashes (like 9b17bb5). Copy one into the commands above.

9 Troubleshooting cheatsheet

You see…What it means & the fix
fatal: not a git repositoryYou're not in a project folder (or it isn't a repo). cd into it, or run git init -b main.
Authentication failedCheck gh auth status; re-login with gh auth login.
Updates were rejected (non-fast-forward)GitHub has changes you don't. git pull --rebase, then git push.
.icloud files / files "missing"Code is in iCloud and got evicted. Move the project to ~/Developer (Section 2).
You're in 'detached HEAD' stateYou checked out a commit, not a branch. git switch main to get back.
Vercel didn't updateDid you push to main? Confirm with git status -sb, then check the Vercel Deployments tab.
When in doubt

git status is your best friend — it tells you what branch you're on, what's changed, and what to do next. Run it often.
